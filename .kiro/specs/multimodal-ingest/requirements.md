# Requirements Document

## Introduction

This document specifies the requirements for a multimodal claims investigator tool that enables claims adjusters to ingest and search across videos, images, audio files, and PDFs using semantic embeddings. The system uses Google Gemini Embedding 2 API to embed different media types into a unified vector space stored in ChromaDB, enabling cross-modal semantic search capabilities.

## Glossary

- **Ingest_Service**: The backend service responsible for accepting media files and processing them for storage
- **Embedding_Service**: The wrapper service that interfaces with Google Gemini Embedding 2 API
- **Vector_Database**: ChromaDB instance storing embeddings and associated metadata
- **Search_Service**: The service that processes search queries and retrieves relevant results
- **Investigation_Service**: The service that uses Gemini 2.5 Pro to reason over retrieved evidence and answer questions
- **Claims_Adjuster**: The end user who uploads files and performs searches
- **Media_File**: A video, image, audio file, or PDF document
- **Evidence_File**: A stored media file associated with a claim that can be analyzed
- **Embedding_Vector**: The numerical vector representation of a media file generated by Gemini Embedding 2
- **Metadata**: Associated information about a media file (filename, type, upload timestamp, etc.)

## Requirements

### Requirement 1: File Ingestion

**User Story:** As a claims adjuster, I want to upload media files to the system, so that I can later search across them semantically.

#### Acceptance Criteria

1. WHEN a claims adjuster uploads a video file, THE Ingest_Service SHALL accept the file and process it for embedding
2. WHEN a claims adjuster uploads an image file, THE Ingest_Service SHALL accept the file and process it for embedding
3. WHEN a claims adjuster uploads an audio file, THE Ingest_Service SHALL accept the file and process it for embedding
4. WHEN a claims adjuster uploads a PDF file, THE Ingest_Service SHALL accept the file and process it for embedding
5. WHEN an unsupported file type is uploaded, THE Ingest_Service SHALL reject the file and return a descriptive error message
6. WHEN a file upload is successful, THE Ingest_Service SHALL return a success response with the file identifier

### Requirement 2: Embedding Generation

**User Story:** As a system, I want to generate embeddings for different media types using a unified API, so that all content exists in the same vector space.

#### Acceptance Criteria

1. WHEN a video file is provided, THE Embedding_Service SHALL call Gemini Embedding 2 API with appropriate video parameters and return the embedding vector
2. WHEN an image file is provided, THE Embedding_Service SHALL call Gemini Embedding 2 API with appropriate image parameters and return the embedding vector
3. WHEN an audio file is provided, THE Embedding_Service SHALL call Gemini Embedding 2 API with appropriate audio parameters and return the embedding vector
4. WHEN a PDF file is provided, THE Embedding_Service SHALL call Gemini Embedding 2 API with appropriate document parameters and return the embedding vector
5. IF the Gemini API returns an error, THEN THE Embedding_Service SHALL propagate a descriptive error message
6. WHEN an embedding is generated, THE Embedding_Service SHALL return a vector of consistent dimensionality regardless of input media type

### Requirement 3: Vector Storage

**User Story:** As a system, I want to store embeddings with their metadata in a vector database, so that they can be efficiently retrieved during searches.

#### Acceptance Criteria

1. WHEN an embedding vector is generated, THE Ingest_Service SHALL store it in the Vector_Database along with associated metadata
2. WHEN storing a vector, THE Ingest_Service SHALL include metadata fields for filename, file type, upload timestamp, and file size
3. WHEN a storage operation completes successfully, THE Ingest_Service SHALL return the unique identifier assigned to the stored vector
4. IF a storage operation fails, THEN THE Ingest_Service SHALL return an error message and not leave partial data in the database

### Requirement 4: Semantic Search

**User Story:** As a claims adjuster, I want to search using natural language text queries, so that I can find relevant media files across all modalities.

#### Acceptance Criteria

1. WHEN a claims adjuster submits a text query, THE Search_Service SHALL embed the query using Gemini Embedding 2 API
2. WHEN a query embedding is generated, THE Search_Service SHALL retrieve the top-k most similar vectors from the Vector_Database
3. WHEN search results are returned, THE Search_Service SHALL include the metadata for each matching media file
4. WHEN search results are returned, THE Search_Service SHALL include a similarity score for each result
5. WHERE a claims adjuster specifies a result limit, THE Search_Service SHALL return at most that number of results
6. WHEN no results meet a minimum similarity threshold, THE Search_Service SHALL return an empty result set

### Requirement 5: API Endpoints

**User Story:** As a frontend developer, I want well-defined REST API endpoints, so that I can integrate the backend services with the React UI.

#### Acceptance Criteria

1. THE Ingest_Service SHALL expose a POST endpoint that accepts multipart form data containing a media file
2. THE Search_Service SHALL expose a POST endpoint that accepts a JSON payload containing a text query and optional parameters
3. WHEN an API request is malformed, THE system SHALL return a 400 status code with a descriptive error message
4. WHEN an API request is successful, THE system SHALL return a 200 status code with the appropriate response payload
5. IF an internal server error occurs, THEN THE system SHALL return a 500 status code with an error message

### Requirement 6: File Validation

**User Story:** As a system administrator, I want uploaded files to be validated, so that the system remains secure and stable.

#### Acceptance Criteria

1. WHEN a file is uploaded, THE Ingest_Service SHALL validate that the file size does not exceed a configured maximum limit
2. WHEN a file is uploaded, THE Ingest_Service SHALL validate the file extension against a whitelist of supported types
3. WHEN a file is uploaded, THE Ingest_Service SHALL validate that the file is not empty
4. IF any validation check fails, THEN THE Ingest_Service SHALL reject the upload and return a specific error message indicating which validation failed

### Requirement 7: Frontend Interface

**User Story:** As a claims adjuster, I want an intuitive web interface, so that I can easily upload files and perform searches without technical knowledge.

#### Acceptance Criteria

1. WHEN a claims adjuster visits the application, THE Frontend SHALL display a file upload area that accepts drag-and-drop and click-to-browse interactions
2. WHEN a claims adjuster visits the application, THE Frontend SHALL display a search box for entering text queries
3. WHEN a file upload is in progress, THE Frontend SHALL display a progress indicator
4. WHEN a search is in progress, THE Frontend SHALL display a loading indicator
5. WHEN search results are returned, THE Frontend SHALL display them in a clear, organized format with metadata visible
6. IF an error occurs during upload or search, THEN THE Frontend SHALL display a user-friendly error message

### Requirement 8: Multimodal Investigation

**User Story:** As a claims adjuster, I want to ask natural language questions about claims and receive AI-powered analysis across all evidence types, so that I can quickly identify inconsistencies and key findings.

#### Acceptance Criteria

1. WHEN a claims adjuster submits a natural language question, THE Investigation_Service SHALL embed the question and retrieve the top-k most relevant evidence files
2. WHEN relevant evidence files are retrieved, THE Investigation_Service SHALL send the question and all evidence files to Gemini 2.5 Pro for multimodal reasoning
3. WHEN Gemini 2.5 Pro analyzes the evidence, THE Investigation_Service SHALL return a plain-English investigator report that directly answers the question
4. WHEN multiple evidence files are analyzed, THE Investigation_Service SHALL identify and report any inconsistencies between files
5. WHERE a claims adjuster specifies a claim_id, THE Investigation_Service SHALL scope the evidence retrieval to that specific claim
6. WHERE a claims adjuster specifies a top_k parameter, THE Investigation_Service SHALL retrieve at most that number of evidence files
7. WHEN evidence is insufficient to answer the question, THE Investigation_Service SHALL explicitly state that in the response
8. WHEN returning an investigation report, THE Investigation_Service SHALL include source references showing which evidence files were analyzed

### Requirement 9: Evidence File Management

**User Story:** As a system, I want to store evidence files on disk and track their paths in metadata, so that they can be loaded for multimodal reasoning.

#### Acceptance Criteria

1. WHEN a media file is ingested, THE Ingest_Service SHALL store the file on disk in an organized directory structure
2. WHEN storing a file, THE Ingest_Service SHALL record the file path in the Vector_Database metadata
3. WHERE a claim_id is provided during ingestion, THE Ingest_Service SHALL include it in the metadata for scoped retrieval
4. WHEN the Investigation_Service retrieves evidence, THE system SHALL load files from disk using the stored paths
5. IF a referenced file cannot be found on disk, THEN THE Investigation_Service SHALL continue processing other evidence and note the missing file

### Requirement 10: Error Handling

**User Story:** As a claims adjuster, I want clear error messages when something goes wrong, so that I understand what happened and can take corrective action.

#### Acceptance Criteria

1. WHEN a network error occurs during file upload, THE Frontend SHALL display a message indicating the connection issue
2. WHEN an unsupported file type is uploaded, THE Frontend SHALL display a message listing the supported file types
3. WHEN a file exceeds the size limit, THE Frontend SHALL display a message indicating the maximum allowed size
4. WHEN the Gemini API is unavailable, THE system SHALL return an error message indicating the embedding service is temporarily unavailable
5. WHEN the Vector_Database is unavailable, THE system SHALL return an error message indicating the storage service is temporarily unavailable
6. IF an investigation request fails, THEN THE Investigation_Service SHALL return a 500 status code with a descriptive error message
